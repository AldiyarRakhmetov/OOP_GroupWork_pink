Student Class (and General Information for Classes in “users” Folder)

Student class is a child of a User class, which means it has an id, username and password; this applies to every other User child. Students have the ability to register for courses, view marks in their marks hashmap, update GPA and set themselves a supervisor if they happen to be researchers.

Students can also view their profiles via the viewProfile() method, which is also a User method overridden by all of its children. Student class also contains getters and setters.

Students themselves are also able to be researchers. This is implemented using two variables: isResearcher and researcherImpl.

The Boolean parameter isResearcher allows researcher-related methods to determine whether they can be used. If the student is not a researcher, related methods will display a message explaining it.

The researcherImpl object handles all researcher-based methods and performs the required logic internally. This implementation also applies to other classes capable of becoming researchers.

Teacher, Employee, TechSupportSpecialist, Admin Classes

Teacher, Employee and TechSupportSpecialist classes (which itself is a child of Employee) are similar to Student class, but each has different available methods.

Teachers are able to give students marks and have a unique TeacherTitle enum value. Employees can message each other and give complaints to other employees. TechSupportSpecialists can receive, accept and progress orders assigned to them.

Additional Notes

  • TechSupportSpecialist cannot be a researcher, therefore its isResearcher value is always false.
  • Teacher throws a CourseRetakeLimitException if a student cannot be given a mark for any reason.
  • Employee cannot clear the complaints vector.

Admin is the most unique class among those mentioned. Although it is also a child of User, it does not contain isResearcher. Admin’s main purpose is to add, remove and check Users under its jurisdiction.

Supporting Classes

While classes inside the “users” folder are the primary ones, there are additional classes that support their functionality.

Enumerators and exceptions are straightforward, but other supporting classes are also important.

Researcher is the only interface in the project and defines how a researcher should function, while ResearcherImpl implements its logic.

Mark, Lesson and CourseRegistration are supplementary classes for the Course class, which is required by Student and Teacher to function properly.

Course can also store Students and Teachers belonging to it inside “teachers” and “students” lists.

Transcript, while located in a different folder, is also an important part of User-related classes.

The folder titled system contains supplementary classes that do not belong to the “course” category:

  • LogEntry — used by Admin to keep track of system logs.
  • News and Report — both can print themselves using unique commands and are used by Manager for handling reports and news.
  • Order — used by TechSupportSpecialist to process orders with statuses defined by the OrderStatus enum.
